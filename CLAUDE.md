# AI 协作工程规范

> 底层思维框架与行为底线。优先级：用户明确指令 > 本规范 > 项目 README.md；代码上下文作为事实来源，不得凭规范覆盖代码事实。

## 🔌 启动协议(必须执行)

本规范由 harness 通过**软链接**分发。Claude Code CLI / Zed IDE 默认不跟随软链接进入目录，**必须执行命令**：

- PowerShell：`Get-ChildItem .claude/skills/ -Force | Select-Object Name`
- Shell：`ls -la .claude/skills/`
- 读取 `README.md`
- 读取 `.context-guard/index.md`
- 读取 `.claude/skills/key-points-mem/references/index.md`

注意：本文件也是软链接，⚠️ 勿就地编辑（软链接自动同步）

## 🏗️ 工程定位

- 工程专属对话沉淀 → `.context-guard/`（按 context-guard 规则评估与写入）
- 通用知识 → `.claude/skills/key-points-mem/references/`（需用户授权）

## 🛡️ 防御性边界

1. **绝对尊重架构**：保护既有架构分层与模块边界，不得擅自重组目录或改动公共接口。
2. **最小侵入**：只改必需代码。遇大规模重构，说明原因和代价，询问是否继续。
3. **禁止隐式破坏**：不得无关增删、覆写现有代码和注释；必要修改必须服务于当前目标，并说明影响。
4. **业务与规范分离**：业务说明仅写入项目 `README.md`，不写入全局 `CLAUDE.md`。
5. **显式路径优先**：用户给出明确文件/目录路径时，直接处理该路径，不得自行跳转到其他目录或仓库。

## 🎯 编码原则

1. **先想后写**：先查找并阅读相关代码及 `README.md`；陈述关键假设，有困惑就问。
2. **极简优先**：50 行能解决不写 200 行。不添加未要求的功能或抽象。
3. **精准修改**：不改相邻代码。不要回滚或覆盖用户已有修改。自己制造的孤儿代码（import/变量）自己清理。
4. **目标驱动**：任务转化为可验证目标，给出分步计划与验证点；修改后执行最相关验证，无法执行则说明原因。
5. **遵循现有风格**：优先沿用项目已有结构、命名、工具、错误处理和测试方式。
6. **参数验证**：文档中的数值/参数必须溯源到对应代码确认一致。

## 🔒 协作红线

**硬约束**（hooks 强制，AI 绕不过）：

- 写入链接文件 → `PreToolUse` 阻断（例外：`.claude/skills/key-points-mem/references/`）
- 文件修改轮次结束时，由 `Stop` hook 触发 context-guard 知识沉淀评估。

**软约束**（AI 自觉遵守）：

- **讨论原则**：日常讨论先结论后依据；debug/排查先展示过程再给结论。避免空泛铺垫和绕圈子。
- **3 次验证**：同一问题验证性命令 ≤3 次。3 次未定位根因 → 停止并请求上下文。
  遇报错先自查再执行：①工具版本/实现是什么？②报错与哪个参数或路径强相关？③最小成功案例和失败案例的差异是什么？
- **防重复**：禁止重复陈述或连续输出相同计划。一旦发生，立即停下——失败/中断时说明原因+下一步（≤3 项），其他情况问用户。
- **提交约束**：没有用户允许不得提交代码
