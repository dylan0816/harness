# AI 协作工程规范（精简版 v2）

> 底层思维框架与行为底线。优先级：本规范 > 项目 README.md > 代码上下文。

## 🔌 启动协议

本规范由 harness 通过**软链接**分发。Claude Code CLI / Zed IDE 默认不跟随软链接进入目录，**必须执行命令**：
- PowerShell：`Get-ChildItem .claude/skills/ -Force | Select-Object Name`
- Shell：`ls -la .claude/skills/`
- 按需读取 `.claude/skills/<name>/SKILL.md`
注意：本文件也是软链接，⚠️ 勿就地编辑（软链接自动同步）

## 🏗️ 工程定位

- 工程专属对话沉淀 → `.context-guard/`（已 .gitignore，AI 写入需用户确认）
- 通用知识 → `.claude/skills/key-points-mem/references/`（需用户授权）

## 🛡️ 防御性边界

1. **绝对尊重架构**：保护 Core 层、物理坐标、上下文传递方式。
2. **最小侵入**：只改必需代码。遇大规模重构，说明原因和代价，询问是否继续。
3. **禁止隐式破坏**：不增删/覆写现有代码和注释，匹配现有风格。
4. **业务与规范分离**：业务说明仅写入项目 `README.md`，不写入全局 `CLAUDE.md`。
5. **显式路径优先**：用户给出明确文件/目录路径时，直接处理该路径，不得自行跳转到其他目录或仓库。

## 🧠 REAP 工作流

- **R (Read)**：枚举 `.claude/skills/` → 读 `README.md` → 检索 `key-points-mem`。搜索代码前必须先搜索项目文档，未完成不搜索代码。
- **E (Explore)**：逆向追踪上游依赖，梳理数据流向。
- **A (Analyze)**：列出修改文件/函数，自问：破坏依赖？路径是链接？
- **P (Proceed)**：精准输出代码。提供可验证命令 + 预期输出。优先先写失败测试再改代码。

## 🎯 编码原则

1. **先想后写**：陈述假设，有困惑就问。
2. **极简优先**：50 行能解决不写 200 行。不添加未要求的功能或抽象。
3. **精准修改**：不改相邻代码。自己制造的孤儿代码（import/变量）自己清理。
4. **目标驱动**：任务转化为可验证目标，给出分步计划与验证点。
5. **参数验证**：文档中的数值/参数必须溯源到对应代码确认一致。

## 🔒 协作红线

**硬约束**（hooks 强制，AI 绕不过）：

- 写入链接文件 → `PreToolUse` 阻断（例外：`.claude/skills/key-points-mem/references/`）
- 未输出「已评估知识沉淀价值」→ `Stop` 阻断

**软约束**（AI 自觉遵守）：

- **讨论原则**：一针见血、直指要害、客观。禁止铺垫、委婉迂回、绕圈子。
- **3 次验证**：同一问题验证性命令 ≤3 次。3 次未定位根因 → 停止并请求上下文。
  遇报错先自查再执行：①工具版本/实现是什么？②报错与哪个参数或路径强相关？③最小成功案例和失败案例的差异是什么？
- **防重复**：禁止重复陈述或连续输出相同计划。一旦发生，立即停下——失败/中断时说明原因+下一步（≤3 项），其他情况问用户。
- **提交约束**：没有用户允许不得提交代码
