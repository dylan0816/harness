# Context Guard

> 优先级：辅助行为，不干扰主任务，沉淀失败不阻塞任务完成。

## 触发条件

1. **主任务收尾阶段**（任务完成前，最后一步执行）

## 价值判定

AI 基于对话事实（非主观陈述）逐项核对以下信号：

1. 代码/配置出现结构变更（新增文件/接口/模块） → 架构决策
2. 持续报错或行为不符合预期，最终定位根因并修复 → 根因分析
3. 建立新的命名规则、错误码格式、接口约定或 lint 规则 → 规范确立
4. 修复中纠正了 AI 或用户对库/语言特性/框架行为的错误理解 → 知识盲区
5. 比较两种以上方案，给出优劣依据并选择其一 → 方案对比
6. 解决不直接报错但破坏构建/运行的问题 → 踩坑记录

**结果**：命中 ≥1 个 → 自动记录；全部未命中 → 跳过，回复"已评估知识沉淀价值（跳过：{理由}）"，必须列出检查了哪些信号。

**不触发**：纯执行性操作（改变量名/加注释/修 typo）、简单问答（函数用法/语法确认/配置路径）。

## 执行流程

### A. 分层决策

命中信号后默认自动记录，写入后在回复末尾告知：
`已自动记录到 .context-guard/<课题>.md，回复 #undo 可撤回`

仅以下情况询问用户（回复"等待确认"）：
- 结论涉及 A/B 方案取舍，AI 无法判断用户偏好
- 用户明确表态"暂定"、"先这样后面再调"等临时结论

询问格式：
> 本次对话涉及 **[信号名]**，建议记录要点。是否记录？（Y/N/补充）

用户 N 或模糊 → 跳过。用户否认清晰命中 → 尊重决定。用户确认后写入。

### B. 摘要

文件以**课题**命名（如 `stop-hook.md`、`deploy-script.md`），课题名用短横线连接。同一课题的新讨论追加到已有文件末尾，不新建文件。

文件顶部必须有**概述**（2-3 句），让 AI 快速判断是否需要继续阅读。

```markdown
# <课题名>

**标签**: #signal-tag #domain-tag

> **概述**: [这个课题是什么、核心结论是什么、当前状态]

---

## YYYY-MM-DD 第N轮 — [本轮小标题]
[对话的关键结论、决策、踩坑经验 — 自然语言，每轮300字以内]
```

每条记录是一个对话片段的总结，不填表，只记录值得回头看的东西。多轮讨论同一课题时逐轮追加，形成演进记录。

### C. 标签规范

每条记录必须包含至少一个信号标签，领域标签按需追加。

**信号标签**（对应六大信号，必选其一）：

| 标签 | 信号 |
|------|------|
| `#architecture` | 架构决策 |
| `#root-cause` | 根因分析 |
| `#convention` | 规范确立 |
| `#knowledge-gap` | 知识盲区 |
| `#decision` | 方案对比 |
| `#pitfall` | 踩坑记录 |

**领域标签**（按实际内容自由添加）：如 `#deploy` `#git` `#bash` `#powershell` `#hook` `#skill` `#transcript` 等。

### D. 写入

1. 确定课题名，创建或追加 `<工程根>/.context-guard/<课题名>.md`（目录不存在则创建）
2. 课题名使用短横线连接（如 `stop-hook`、`deploy-script`），保持简短
3. 若文件不存在，创建时写入标题和标签行；若已存在，在末尾追加新一轮的 `## YYYY-MM-DD` 小节
4. 同步更新 `<工程根>/.context-guard/index.md`：
   - 课题已有 → 更新该行的「最后更新」日期
   - 课题新增 → 在索引表末尾追加一行

```markdown
| <课题名> | #tag1 #tag2 | YYYY-MM-DD |
```

索引表以 `<!-- INDEX_START -->` 和 `<!-- INDEX_END -->` 标记边界，在这两行之间操作。如果 index.md 不存在，创建时写入表头和标记：

```markdown
# Context Guard 索引

<!-- INDEX_START -->
| 课题 | 标签 | 最后更新 |
|------|------|----------|
<!-- INDEX_END -->
```

### E. 检索

AI 在需要查找历史记录时：

1. 先读 `index.md`（极短，一次读完），按标签/课题名定位候选文件
2. 再读对应 `<课题名>.md`，该文件包含此课题的完整演进记录
3. 不逐文件全量读取

## 与 key-points-mem 的协作

| | key-points-mem | context-guard |
|---|---|---|
| 位置 | `.claude/skills/.../references/` | `<工程根>/.context-guard/` |
| 内容 | 精炼知识点（现象-原因-解决-经验） | 对话上下文+结论，按课题归档 |
| 范围 | 跨工程通用 | 工程专属 |
| 触发 | 用户授权后 AI 提炼 | 自动评估 + 自动记录 |
| 索引 | `references/index.md` | `.context-guard/index.md` |

两个系统独立运行，互不查询。提炼由用户手动发起，AI 不做自动计划。

## 边界

1. **不阻塞主任务**：收尾阶段执行，不打断主流程。
2. **自动记录可撤回**：`#undo` 撤回最近一次自动写入的内容。
3. **上下文保护**：预估剩余不足 500 token 时跳过。
4. **工程隔离**：不跨工程读/写。
5. **清理**：用户自行决定，AI 不主动提议。

## 任务结束标志

Stop hook 检测本轮文件变更后强制要求评估。AI 无需预设格式——hook 阻断时会告知具体写法。
